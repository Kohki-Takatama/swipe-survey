# Daily Check-in プロジェクト

## プロジェクト概要

毎日の体調・習慣を記録するチェックインアプリ。
単一HTMLファイル + GAS Web App という構成で、スマホのブラウザから使用する。

## 現在のファイル構成

| ファイル | 役割 |
|---|---|
| `index.html` | フロントエンド本体（設問UI・送信ロジック） |
| `gas-doPost.gs` | GAS バックエンド（スプレッドシートへの記録） |

## 方向性

**「リリースを目指しつつ、まずは実際の生活にチューニングする」**

- 自分が使い続けられる設問・UXに育てることを優先する
- 並行して、将来のリリース・マネタイズを見据えた設計判断を行う

## 開発方針

### マネタイズを考慮する
将来的な収益化ポイントを意識しながら機能を追加する。
検討候補：
- プレミアムプラン（設問数・設問タイプの上限解放）
- テンプレートマーケット（設問セットの販売・配布）
- チーム・家族プラン（データの共有・比較）
- データエクスポート・分析ダッシュボード（有料機能）

機能追加の際は「これは無料範囲か、有料差別化ポイントか」を意識する。

### 疎結合・コンポーネント化を意識する
メンテナンス性・カスタマイズ性のために、責務を分離して実装する。

- **設問定義（`QUESTIONS`）** はデータとして独立させ、UIロジックに混在させない
- **設問タイプ（swipe / dial / tap）** は追加・差し替えできるよう独立したビルダー関数として実装する
- **送信ロジック・バックエンド** はフロントと切り離し、差し替えやすくする
- 将来的にフレームワーク（React等）へ移行しやすい構造を心がける

## 技術スタック（現状）

- **フロントエンド**: 単一HTMLファイル（Vanilla JS）
- **バックエンド**: Google Apps Script（Web App）
- **データストア**: Google スプレッドシート

## Claude 開発ルール

### ブランチへの push 前に必ず確認すること
ブランチに push する前に、**そのブランチが既に main にマージ済みでないか**を確認すること。
確認コマンド: `gh pr view <PR番号> --json state,mergedAt` または `git log --oneline origin/main | grep <ブランチ名>`
マージ済みのブランチに push すると意図しない履歴汚染が起きる。マージ済みの場合は新しいブランチを切ること。
